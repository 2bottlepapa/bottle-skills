# Skill Explanation

根据技能文档生成一份面向使用者的说明文档。

## 目标

当用户提供一个技能文档时，整理成一份 Markdown 说明文档，固定包含以下三部分：

1. 这个技能的主要功能
2. 它依赖的技术或能力实现方式
3. 使用者可以如何触发使用它

## 触发条件

仅在以下情形触发：

- 用户明确要求“根据文档解释这个技能，输出文档”
- 用户明确要求“根据 SKILL.md 生成说明文档”
- 用户明确要求“把这个技能文档整理成一份说明文档”
- 用户提供技能文档路径、当前打开的技能文档，或公开 GitHub 技能文档链接，并要求生成说明文档

以下情形不要触发本技能：

- 用户只是询问某个技能的功能
- 用户只是询问某个技能怎么触发
- 用户只要简短说明，不要求生成文档

## 推荐触发提示词

以下写法可稳定触发本技能（同时包含”技能文档”和”输出文档”两个意图）：

- 根据这个 SKILL.md 生成一份技能说明文档
- 把这个技能文档整理成一份给使用者看的说明文档，输出到默认目录
- 根据这个技能文档生成说明文档：`<本地路径 或 GitHub 链接>`

以下写法不触发（只是口头问答，无”生成文档”意图）：

- 这个技能是干嘛的
- 这个技能怎么用

## 支持的输入来源

本技能支持 3 种输入来源，按以下顺序识别：

### 1. 本地文件路径

如果用户提供本地技能文档路径，直接读取该文件。

优先识别：

- `SKILL.md`
- 其他看起来是技能说明的 Markdown 文档

### 2. 当前打开的技能文档

如果用户没有提供文件路径，但当前编辑器中打开的是技能文档，则使用当前打开文件作为输入。

可接受：

- 文件名为 `SKILL.md`
- 文件内容明显包含技能 frontmatter 或技能说明结构

### 3. 公开 GitHub 仓库中的技能文档链接

仅支持公开 GitHub 仓库链接，私有仓库不支持。

优先支持以下两类链接：

- 标准 blob 链接，例如：`https://github.com/<owner>/<repo>/blob/<branch>/SKILL.md`
- raw 链接，例如：`https://raw.githubusercontent.com/<owner>/<repo>/<branch>/SKILL.md`

处理规则：

1. 如果是 `github.com/.../blob/...` 链接，先转换成 `raw.githubusercontent.com` 原始文档链接
2. 如果已经是 `raw.githubusercontent.com` 链接，则直接读取
3. 如果不是公开仓库、不是标准 blob/raw 技能文档链接，停止并告诉用户当前版本不支持该 GitHub 输入形式

不要为了支持 GitHub 页面文件预览去解析复杂网页 DOM。第一版只处理公开仓库中的原始技能文档读取。

## 默认输出目录配置

配置文件固定保存在本技能目录下：

- `config.json`

**定位方式：** 通过本 SKILL.md 文件的所在目录推导技能目录。例如，若 SKILL.md 路径为 `.agents/skills/bottle-skills/skill-explanation/SKILL.md`，则 config.json 路径为 `.agents/skills/bottle-skills/skill-explanation/config.json`。

配置项：

- `defaultOutputDir`: 默认输出目录

### 首次使用

首次使用时，先读取 `config.json`。

如果 `defaultOutputDir` 为空或不存在，必须先询问用户：

- “说明文档默认输出到哪个目录？”

只询问这一项，不要扩展成更多偏好问题。

拿到目录后：

1. 将其保存到 `config.json`
2. 告知用户该目录已保存为默认输出目录
3. 再继续生成说明文档

### 后续使用

如果用户本次没有指定新的输出路径，则直接使用 `config.json` 中保存的 `defaultOutputDir`。

如果用户本次明确指定了新的输出路径，则本次按新路径输出；如用户同时表达“以后默认用这个路径”，再更新 `config.json`。

## 输出文件命名规则

根据技能文档内容尽量提取技能名，优先顺序如下：

1. frontmatter 中的 `name`
2. 一级标题或明显的技能标题
3. 若无法可靠提取，则视为未拿到技能名

命名规则：

- 能拿到技能名：`<技能名>-clearer.md`
- 拿不到技能名：`skill-explanation.md`

文件名应做基本清洗：

- 优先使用 frontmatter `name` 字段值（通常为英文，如 `skill-explanation`），直接用于文件名
- 若 `name` 字段含中文，保留原文，不做拼音转换
- 去掉路径分隔符（`/`、`\`）和明显非法字符（`:`、`*`、`?`、`"`、`<`、`>`、`|`）
- 空格替换为连字符 `-`

## 禁止覆盖已有文件

绝对不要覆盖已有文件。

如果目标文件名已存在，则自动追加递增序号：

- `<技能名>-clearer-2.md`
- `<技能名>-clearer-3.md`
- `skill-explanation-2.md`

直到找到不存在的文件名再写入。

## 文档写作要求

生成的说明文档必须是 Markdown，内容要面向普通使用者，避免只复述原文。

输出结构固定为（`补充理解` 为可选节，无内容时省略）：

```md
# <技能名> 技能说明

## 1. 功能

## 2. 实现方式

## 3. 触发方式

<!-- 以下为可选节，若无补充内容则不输出 -->
## 补充理解
```

写作要求：

- 用清晰、归纳后的语言表达，不机械复制技能原文
- 功能部分写清这个技能解决什么问题、适合什么场景
- 实现方式部分写清它依赖哪些工具、能力、流程或外部服务；若技能为纯提示词驱动、不依赖任何外部工具，则写明”基于模型语言理解与生成能力实现，无外部工具依赖”
- 触发部分写清用户通常可以怎么提需求来触发它
- 如果技能文档中有边界条件或不适用场景，补充到”触发”或”补充理解”中
- “补充理解”专门写：不适用场景、使用限制、易混淆点、注意事项；若无此类内容，省略该节

## 处理流程

### Step 1: 确认是否应触发本技能

先判断用户是不是明确要求“输出文档”。

如果不是，则不要触发本技能，直接正常回答用户问题。

### Step 2: 获取技能文档内容

按以下优先顺序处理：

1. 用户明确给出本地文件路径
2. 使用当前打开的技能文档（文件名为 `SKILL.md` 或内容包含技能 frontmatter）
3. 用户明确给出公开 GitHub 技能文档链接

如果三者都没有，要求用户补充输入来源。

### Step 3: 读取并解析技能文档

从技能文档中提取这些关键信息：

- 技能名
- description 或简介
- allowed-tools 或依赖工具
- workflow、usage、prerequisites、output、limits 等结构化信息
- 明确的触发词、触发场景、非触发场景

### Step 4: 确定输出目录

1. 读取本技能目录下的 `config.json`
2. 若没有默认输出目录，则询问用户并保存
3. 若用户本轮显式指定路径，则优先使用该路径

### Step 5: 生成输出文件名

按命名规则生成目标文件名，并检查是否重名。

如果重名，自动递增追加序号，直到生成唯一文件名。

### Step 6: 写入说明文档

将整理后的 Markdown 写入目标目录。

完成后告诉用户：

- 使用了哪份技能文档作为输入
- 输出到了哪个目录
- 最终文件名是什么

## 失败处理

### 无法读取输入

如果本地文件不存在、当前打开文档不是技能文档、或 GitHub 链接不受支持，应明确说明原因，并要求用户提供可读取的技能文档。

### GitHub 链接不受支持

如果是私有仓库链接，或不是公开仓库中的标准 blob/raw 技能文档链接，直接说明当前版本不支持，并建议用户改为：

- 提供本地文件路径
- 打开技能文档作为当前文件
- 提供公开仓库中的原始技能文档链接

### 无法提取技能名

继续执行，输出文件名退化为 `skill-explanation.md`，不要因此中断。

## 约束

- 只在用户要求生成文档时触发
- 只询问默认输出目录这一项偏好
- 默认配置固定保存在本技能目录下
- 仅支持公开 GitHub 仓库链接
- 不覆盖已有文件
- 输出必须是 Markdown 文档
